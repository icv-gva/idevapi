# Exploration: idevapi-core-jq-path-fix

## Current State

`idevAPI_core.js` (2775 lines) uses a 3-layer resolver chain (`detectLoadFamily` → jsdelivr / GVA / local) to compute `urlAPI` from the script's own `srcCore` URL. Once resolved, `urlAPI` is the base for ALL library loads: jQuery, Leaflet, CSS, images, widgets, i18n files, marker icons.

### The Bug

`resolveLocal` (lines 45–50) correctly matches local paths via regex, but **discards the captured group** and returns a hardcoded `"../../idevapi"`:

```javascript
function resolveLocal(srcCore) {
    var m = srcCore.match(/^(.*\/)idevapi\/js\/idevAPI_core/);
    if (m) return { urlAPI: "../../idevapi", URLVersion: "" };  // ← HARDCODED
    return null;
}
```

This works for visors at `_visores_tester/<visor>/` (2 levels deep → `../../idevapi` resolves to `/idevapi/`) but **BREAKS for help examples** at `ayuda/ejemplos/1.3/` (3 levels deep → `../../idevapi` resolves to `/ayuda/idevapi/`, which does not exist).

The **fallback** in `detectLoadFamily` (line 24) also hardcodes `"../../idevapi"`:
```javascript
return { urlAPI: "../../idevapi", URLVersion: "" };
```

## Affected Areas

- **`js/idevAPI_core.js`** lines 45–50 — `resolveLocal` function (PRIMARY FIX)
- **`js/idevAPI_core.js`** line 24 — fallback in `detectLoadFamily` (SECONDARY FIX)
- **`js/idevAPI_core.js`** line 79 — `IDEVAPIVersion` bump to `"1.3.22"`

- **`js/idevAPI_tabla.js`**, **`js/idevAPI_filtro.js`**, **`js/idevAPI_config.js`** — use global `urlAPI`, benefit automatically

All consumers use `prot + urlAPI`, which expects `urlAPI` to be EITHER:
- Protocol-relative (`//host/path` — CDN/GVA)
- Page-relative (`../../...` — local, CURRENT, BUGGY)
- OR root-relative (`/path` — would work for ALL depths)

## Approaches

### Approach A: Compute root-relative path from `srcCore` using anchor element (RECOMMENDED)

Replace `resolveLocal` to extract the pathname from `srcCore` and strip `/js/idevAPI_core.*`. Produces a root-relative path like `/idevapi` that is depth-independent.

```javascript
function resolveLocal(srcCore) {
    var a = document.createElement('a');
    a.href = srcCore;
    var p = a.pathname;                          // e.g. "/idevapi/js/idevAPI_core.js"
    var base = p.replace(/\/js\/idevAPI_core.*$/, "");  // e.g. "/idevapi"
    if (base !== p) return { urlAPI: base, URLVersion: "" };
    return null;
}
```

- **Pros**: Depth-independent (works for 2, 3, N levels). Uses browser's native URL parser (IE11+). Handles any server path prefix (not just `/idevapi`). Clean, 5-line change.
- **Cons**: Creates an anchor element (trivial overhead). Requires `srcCore` to be an absolute or resolvable URL (already guaranteed by `getElementsByTagName("script")`).
- **Effort**: Low

### Approach B: Fix regex capture + strip protocol

```javascript
function resolveLocal(srcCore) {
    var m = srcCore.match(/^(?:https?:\/\/[^\/]+)?(\/.*\/)idevapi\/js\/idevAPI_core/);
    if (m) return { urlAPI: m[1] + "idevapi", URLVersion: "" };
    return null;
}
```

- **Pros**: Pure regex, no DOM API. Single line change.
- **Cons**: Fragile regex (non-obvious protocol skipping). Harder to maintain. Doesn't handle edge cases like `file://` or IP addresses with unusual formats.
- **Effort**: Low

### Approach C: Compute relative depth dynamically

Count `/` segments from page to script to calculate the correct `../` count.

- **Pros**: Theoretically precise.
- **Cons**: Complex, fragile, over-engineered. Pointless when a root-relative path exists.
- **Effort**: Medium

### Recommendation: Approach A

Root-relative paths are the standard solution for depth-independent resource loading. The `<a>` element approach uses the browser's own URL parser, which handles edge cases correctly. The resulting `urlAPI` (`"/idevapi"`) works with the existing `prot + urlAPI` pattern: `"http:" + "/idevapi"` → `"http:/idevapi"` resolves to `http://<origin>/idevapi` per RFC 3986.

## Risks

1. **`esbuild` target `es2015`**: The `<a>` element and `pathname` property are available since IE6. es2015 target is safe. No compatibility concern.
2. **Fallback path `/idevapi`**: If the script is loaded from an unexpected subpath (e.g., `/vendor/libs/idevapi/js/...`), the fallback `/idevapi` would be wrong. However, `resolveLocal` should match any path containing `/idevapi/js/idevAPI_core` before the fallback is reached. The fallback is only a last resort safety net.
3. **CDN paths unchanged**: `resolveJsdelivr` and `resolveGVA` are untouched. The fix only affects local loading. CDN consumers see zero change.
4. **Regenerated minified files**: `npm run build` regenerates ALL minified files (20 files). Files whose sources didn't change will produce identical hashes. Only `idevAPI_core-min.js` and its `.map` will have new content.
5. **`IDEVAPIVersion` cache busting**: Bumping to 1.3.22 adds `?v=1.3.22` to library URLs. This is correct for a new release — ensures browsers don't use stale cached libraries.
